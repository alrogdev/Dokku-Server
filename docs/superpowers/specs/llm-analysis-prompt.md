# Промпт для анализа спецификации безопасности Dokku PaaS

**Для:** GLM-5, MiMo V5 Pro, или других LLM-агентов  
**Цель:** Технический аудит спецификации архитектуры безопасного PaaS

---

## Контекст проекта

- **VPS:** SpaceWeb, Ubuntu 24.04 LTS
- **Ресурсы:** 2 CPU, 4 GB RAM, 45 GB NVMe
- **IP:** 80.93.63.169
- **Домен:** clawtech.ru (с wildcard *.clawtech.ru)
- **Назначение:** Pet-projects (Node.js/Express, Vue, React, FastAPI, MCP)
- **Базы данных:** PostgreSQL, SQLite, Redis
- **Требования безопасности:**
  - CIS Ubuntu 24.04 Level 1 compliance
  - CrowdSec с максимальным количеством коллекций
  - File Integrity Monitoring (AIDE)
  - CIS auditing (Lynis)
  - Telegram-отчёты каждые 6 часов

---

## Задача анализа

Проведи технический аудит спецификации архитектуры. Проверь следующие аспекты:

### 1. Безопасность

- [ ] Достаточно ли мер защиты для VPS с публичным IP?
- [ ] Корректна ли стратегия firewall (nftables вместо UFW для избежания конфликтов с CrowdSec)?
- [ ] Правильно ли подобраны CrowdSec коллекции (13 шт.)?
- [ ] Адекватны ли bouncers (nginx + nftables firewall)?
- [ ] Оптимальна ли политика банов SSH (порт 22 + CrowdScenarios)?
- [ ] Правильно ли настроен whitelist (owner IPv4/IPv6 + agent IP)?
- [ ] Достаточно ли защиты для Docker/Dokku окружения?

### 2. Архитектура

- [ ] Оптимален ли выбор инструментов (Ansible + shell scripts)?
- [ ] Корректна ли структура проекта (/ansible, /scripts, /docs, /keys)?
- [ ] Нет ли конфликтов между компонентами (CrowdSec + nftables + Docker)?
- [ ] Правильно ли организована иерархия документации?
- [ ] Адекватны ли права доступа для агента (AGENT.md)?

### 3. Операционная эффективность

- [ ] Реалистично ли расписание cron-задач?
  - CrowdSec summary: каждые 6 часов
  - AIDE FIM: 02:00 MSK ежедневно
  - Lynis audit: воскресенье 03:00 MSK
- [ ] Не будет ли перегрузки при одновременном запуске задач?
- [ ] Достаточно ли ресурсов (4 GB RAM) для всех компонентов?
- [ ] Корректно ли рассчитано потребление диска (AIDE база, логи)?

### 4. Пробелы и риски

- [ ] Что упущено в плане безопасности?
- [ ] Какие attack scenarios не учтены?
- [ ] Есть ли single points of failure?
- [ ] Что произойдёт при компрометации Telegram бота?
- [ ] Предусмотрено ли восстановление после сбоя?
- [ ] Есть ли стратегия обновлений (security patches)?

### 5. Соответствие требованиям

- [ ] Полностью ли покрыты требования CIS Level 1?
- [ ] Корректно ли интегрирован CrowdSec CAPI (Community Blocklist)?
- [ ] Достаточно ли мониторинга (FIM + audit + телеграм)?
- [ ] Правильно ли организована отчётность?

### 6. Улучшения

- [ ] Что можно оптимизировать?
- [ ] Какие best practices пропущены?
- [ ] Что избыточно и можно упростить?
- [ ] Какие современные альтернативы стоит рассмотреть?

---

## Формат ответа

Предоставь структурированный отчёт:

### 🔴 Критические замечания (блокеры)
Проблемы, которые делают архитектуру непригодной для production.

### 🟡 Предупреждения (нужно учесть)
Потенциальные риски или недостатки, требующие внимания.

### 🟢 Рекомендации (улучшения)
Предложения по оптимизации и усилению архитектуры.

### 📊 Общая оценка
- **Проходит / Не проходит** для production pet-projects
- Оценка по шкале 1-10
- Ключевые сильные стороны
- Главные слабости

### 💡 Priority Actions
Топ-3 действия, которые стоит сделать в первую очередь.

## Дополнительные вопросы для размышления

1. **Fail2ban vs CrowdSec:** Нужен ли fail2ban как fallback?
2. **Backup strategy:** Достаточно ли локальных бэкапов или нужен off-site?
3. **Log retention:** Как долго хранить логи для compliance и расследований?
4. **Secret management:** Достаточно ли безопасно хранение ключей в /keys/?
5. **Disaster recovery:** Какое время восстановления (RTO/RPO) приемлемо?
